## 一、概述

- **做了什么**：对 TimerFocus 前端进行全面响应式布局重构，修复了页面不跟随窗口大小变化的问题，补充了缺失的页面导航，并优化了所有组件的自适应行为。
- **为什么做**：原有 UI 使用固定像素宽度（如 340px 侧边栏、420px 统计面板），最大化浏览器窗口后大半空白；设置页面无返回首页按钮；导航栏只有 Logo 和设置入口，体验割裂。
- **涉及模块**：全局样式、5 个视图页面、11 个组件、1 个工具模块
- **最终达到的效果**：
  - 页面 100% 利用窗口宽度，无空白浪费
  - 导航栏新增 5 个页面链接（计时/历史/统计/标签/设置），带当前页高亮
  - 设置页增加返回首页按钮
  - 所有图表使用 ResizeObserver 监听容器变化而非仅 window resize
  - 多重断点适配桌面 → 平板 → 手机

## 四、关键设计决策

### 1. 响应式布局策略：桌面优先 + clamp() + max-width 容器

#### 为什么这样设计

采用"桌面优先"（Desktop First）策略，先确保大屏体验完美，再用 `@media` 向下适配平板和手机。每个页面使用 `max-width` 约束内容宽度并 `margin: 0 auto` 居中，防止超宽屏内容松散。侧边栏使用 CSS `clamp(280px, 26%, 380px)` 实现弹性的宽度，根据视口自动缩放。

#### 替代方案

考虑过使用百分比布局或完全 fluid 布局。但 TimerFocus 作为桌面生产力工具，主要场景是大屏使用，桌面优先更合适。百分比布局在大屏上会导致计时面板过度拉伸，`max-width` + 居中方案更好。

#### 影响

页面从固定 1100px 左右的有效区域扩展到全窗口可用。侧边栏在 1280px 屏宽下约为 330px，在 1920px 屏宽下约为 380px，主内容区域始终占 70%+。

### 2. 导航栏重构：内嵌导航链接代替汉堡菜单

#### 为什么这样设计

原设计导航栏只有 Logo + 设置按钮，用户需要在 URL 栏手动输入路径才能导航。新设计在导航栏中间增加 5 个图标+文字的链接，点击即可切换页面。使用 Vue Router 的 `<router-link>` 组件自动管理 `active` 状态。

当屏幕宽度 < 900px（平板竖屏及以下）时导航链接自动隐藏，仅保留 Logo 和右侧工具栏，避免换行。

#### 替代方案

考虑过使用侧边栏或底部 TabBar。但导航项只有 5 个，顶部一行足够容纳。侧边栏会进一步减少主内容区宽度，不采用。

#### 影响

显著提升页面间导航效率，用户无需记忆路由路径。

### 3. 图表自适应：ResizeObserver 替代 window resize 监听

#### 为什么这样设计

原设计仅监听 `window.resize` 事件触发 `chart.resize()`。但当侧边栏宽度变化、窗口最大化/恢复时，布局重排不会触发 window resize 事件，导致图表尺寸与容器不匹配。改用 `ResizeObserver` 监听图表容器本身的大小变化，任何布局变化都能自动重绘。

#### 替代方案

可同时监听 window resize 和侧边栏宽度变化。但 ResizeObserver 是浏览器原生 API（IE 以外全支持），一个 API 覆盖所有场景，更简洁。

#### 影响

图表始终与容器保持一致，消除了布局变化后图表不变的问题。

### 4. 每个页面的返回导航

#### 为什么这样设计

原设计中设置、历史、统计、标签等页面没有任何返回机制，用户只能手动修改 URL 回到首页。现在每个子页面都在标题左侧添加了 `← 返回` 或 `返回首页` 按钮，使用 Vue Router 导航。

#### 影响

修复了"进设置后出不来"的核心体验问题。

## 五、问题与解决方案

### 问题1：最大化窗口后页面有大片空白

#### 现象

窗口最大化到 1920x1080 后，页面内容仅占据约 1100px 宽度，右侧和底部大量空白。

#### 根因

1. 全局样式 `#app { overflow: hidden }` 限制了布局
2. 侧边栏固定 `width: 340px` 不随视口缩放
3. 统计页侧面板固定 `420px` 不缩放
4. 内容区无 `max-width` 约束，但 flex 容器受子元素影响未扩展

#### 解决方案

- 全局 `width: 100vw; max-width: 100%` 确保占满视口
- 侧边栏改用 `clamp(280px, 26%, 380px)` 根据视口弹性缩放
- 各视图添加 `max-width` 居中容器，保证大屏下内容宽度合理
- 图表区改用 `flex: 1; min-height: 0` 自适应剩余空间

#### 最终结果

页面在任何分辨率下都充分利用可用空间，无溢出的空白区域。

### 问题2：设置页无法返回首页

#### 现象

用户点击"设置"进入设置页面后，找不到任何返回主页面的按钮或链接。

#### 根因

1. AppHeader 中没有主页导航链接（只有 Logo，但不是可点击的链接）
2. SettingsView 没有返回按钮

#### 解决方案

- AppHeader 新增完整的导航栏，包含 5 个页面链接
- SettingsView 顶部新增"← 返回首页"按钮
- 所有子页面（History、Stats、Tags）同样添加返回按钮

#### 最终结果

任何页面都可以通过导航栏或返回按钮回到首页。

### 问题3：图表在布局变化时不重绘

#### 现象

手动拖动调整浏览器宽度时图表响应正常，但从窗口化切换到最大化或侧边栏展开/收起时图表大小不变。

#### 根因

图表仅监听 `window.resize` 事件，对 flex/grid 布局变化导致的容器尺寸变更不感知。

#### 解决方案

将所有图表组件的 resize 监听从 `window.addEventListener('resize', ...)` 改为 `new ResizeObserver(...).observe(chartContainer)`。

#### 最终结果

图表始终跟随容器大小实时更新。

## 十、开发复盘

### 本次最大收获

CSS `clamp()` 函数在响应式布局中非常实用，一个声明即可覆盖桌面到平板的宽度范围，无需多个 `@media` 查询。`ResizeObserver` 比 `window.resize` 精确得多，是现代 Web 应用的图表自适应的标准做法。

### 本次踩过的坑

Element Plus 的 `el-row` / `el-col` 响应式类名（`:xs`, `:sm`, `:md` 等）在部分组件中表现不一致，需要 `:deep()` 穿透样式覆盖。后续应尽量用原生 CSS Grid/Flexbox 替代 Element Plus 的栅格系统。

### 如果重新设计

会从一开始就建立完整的响应式断点体系和容器系统，而不是后期"打补丁"。每个视图页面都应先设定 `max-width` + `margin: auto` 居中框架，再填充内容。
